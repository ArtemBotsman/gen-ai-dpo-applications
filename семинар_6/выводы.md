# Выводы · Семинар 6 · PWC на прочность

## 1. Валидатор схемы

Сигнатура:

```python
VALID_TOOLS = {"get_fx_rate", "get_key_rate", "get_inflation", "calculate"}

def validate_plan(plan: Plan) -> list[str]:
    """Вернуть список ошибок плана (пустой — всё ок)."""
```

Проверяет два типа ошибок:
- **выдуманный инструмент** в `expected_tools` (например `get_cumulative_inflation`);
- **битый `depends_on`** — ссылка на несуществующий id или на самого себя.

Встроен в `run_pwc()` сразу после `planner()`: при ошибках планировщик вызывается повторно с `feedback`.

**Примеры пойманных планов:**

| Вопрос | Что предложил Планировщик | Ошибка валидатора |
|--------|---------------------------|-------------------|
| Q3: накопленная инфляция 2022–2026 | `get_cumulative_inflation` в подвопросе 1 | `неизвестный инструмент 'get_cumulative_inflation'` |
| Q6: ставка vs ИПЦ г/г | `get_real_rate` или пустой `depends_on` на id 99 | неизвестный tool / `depends_on ссылается на id N, которого нет` |

После фикса на Q3 PWC+валидатор в 3/3 прогонах либо перепланировал на честный план с `calculate` по помесячному ИПЦ, либо вернул пустой план с объяснением «одним вызовом не решить» — оба варианта засчитываются как корректное поведение.

Проверка из ДЗ:

```bash
python orchestrator.py "Какова накопленная инфляция с января 2022 по март 2026?"
```

## 2. Параллельность

Реализация: `_topological_levels()` + `execute_level()` с `ThreadPoolExecutor`.

| Вопрос | Последовательно | Параллельно | Ускорение |
|--------|-----------------|-------------|-----------|
| Q1 (2 независимых `get_fx_rate`) | 28.01 с | 24.44 с | **1.15×** |
| Q4 (3 независимых `get_fx_rate`) | 27.23 с | 29.81 с | 0.91× |

На Q1 два независимых запроса курсов реально ускоряются. На Q4 выигрыш съедается сетевой латентностью и накладными расходами — при 3 параллельных HTTP-вызовах к LLM+tools разница в пределах шума. Имеет смысл параллелить только уровни с 2+ тяжёлыми независимыми подвопросами.

```bash
python orchestrator.py --benchmark "Во сколько раз USD подорожал с 1 января 2022 по сегодня?"
```

## 3. Замер критики (угодливость)

5 заведомо битых кейсов × 10 прогонов при T=0.0 и T=0.7 (`python critic_sycophancy.py`):

| Битый кейс | T=0.0, ложных принятий | T=0.7, ложных принятий |
|------------|------------------------|------------------------|
| арифметика без calculate | 9/10 | 8/10 |
| выдуманное число | 10/10 | 10/10 |
| несогласованные данные | 0/10 | 0/10 |
| ответ с ошибкой инструмента | 0/10 | 0/10 |
| план не покрывает вопрос | 0/10 | 0/10 |

**Гипотеза «шум при T=0.7 лечит зеркальное соглашение» не подтвердилась:** на арифметике без `calculate` ложных принятий даже чуть меньше при 0.7 (8 vs 9), а на выдуманном числе Критик принимает битый ответ в 10/10 при обеих температурах. Зато явные логические несостыковки (разные цифры между подвопросами, ошибка инструмента, неполный план) он ловит стабильно при любой T.

## 5. Вывод: когда PWC оправдан

**PWC имеет смысл**, когда:
- вопрос требует явной декомпозиции и разных наборов tools на подзадачи (Q4, Q5);
- нужен валидатор схемы против галлюцинаций инструментов (Q3);
- есть независимые подвопросы, которые можно распараллелить (Q1, умеренный выигрыш ~15%).

**Одиночный агент лучше**, когда:
- задача укладывается в 3–5 шагов ReAct без декомпозиции (Q1, Q2, Q5 — single стабильнее);
- накладные расходы PWC (3 LLM-роли × итерации) не окупаются точностью;
- Критик не добавляет надёжности — на «правдоподобных» битых ответах он почти не работает.

Главный урок: **PWC без жёсткого валидатора и без строгого Критика** не сильнее single-agent, а иногда хуже из-за накопления ошибок между ролями. Валидатор схемы — обязательный слой; Критик на одной модели слишком «угодлив» к числам, которые звучат правдоподобно.
