## 1. Новый инструмент `compare_periods`

Сигнатура:

```python
def compare_periods(
    metric: str,      # key_rate | fx_USD | fx_EUR | fx_CNY | cpi | unemployment
    period_a: str,    # YYYY-MM или YYYY-MM-DD
    period_b: str,
) -> dict
```

Внутри вызывает `get_fx_rate`, `get_key_rate`, `get_inflation`, `get_unemployment` — логику не дублировал.

**Примеры, где агент его зовёт:**

| Запрос | Вызов | Результат |
|--------|-------|-----------|
| «Во сколько раз вырос USD с января 2022 по апрель 2026?» | `compare_periods(fx_USD, 2022-01, 2026-04)` | ratio ≈ 1.09 |
| «На сколько п.п. изменилась ключевая ставка с марта 2022 по июнь 2024?» | `compare_periods(key_rate, 2022-03, 2024-06)` | delta = −4 п.п. |
| «Как изменился курс юаня за 2024 год?» | `compare_periods(fx_CNY, 2024-01, 2024-12)` | delta ≈ +2.15 руб. |

Проверка из ДЗ:

```bash
python agent.py "Во сколько раз вырос курс USD с января 2022 по апрель 2026?"
```

Агент за 2 шага: `compare_periods` → финальный ответ «в 1,09 раза».

## 3. Диагностика — 3 типа ошибок из trace.jsonl

### Ошибка 1 · Инструмент вернул error (нет данных)

**Тип:** инструмент упал / вернул `{"error": ...}`

**Трасса:**
```json
{"step": 1, "call": "get_inflation", "args": {"year": 2026, "month": 5}, "obs": {"error": "нет данных ИПЦ на 2026-05"}}
{"step": 2, "call": "get_inflation", "args": {"year": 2026, "month": 4}, "obs": {"error": "нет данных ИПЦ на 2026-04"}}
```

**Причина:** CSV с инфляцией заканчивается на март 2026, а агент запрашивает май/апрель 2026 для «текущей» инфляции.

**Что бы исправил:** в системном промпте явно писать «последний доступный месяц в CSV — март 2026» или в `get_inflation` возвращать ближайший месяц с пометкой `stale`.

---

### Ошибка 2 · Исчерпан лимит шагов (max_iter)

**Тип:** бесконечный / незавершённый цикл (обрезан лимитом)

**Трасса:** прогон с `--max-iter 1` на вопрос про индекс нищеты → `error: исчерпан лимит шагов max_iter=1`. Агент успел только вызвать инструменты, но не выдал финал.

**Причина:** сложный multi-hop не укладывается в 1 итерацию — нужно минимум 3–5 шагов.

**Что бы исправил:** для составных вопросов держать `max_iter=8` (как сейчас) или добавить в промпт подсказку «сначала собери все числа параллельно».

---

### Ошибка 3 · Неправильный порядок / лишние вызовы

**Тип:** неправильный порядок вызовов / избыточные инструменты

**Трасса (Q10):**
```json
{"step": 1, "call": "compare_periods", "args": {"metric": "fx_CNY", "period_a": "2024-01", "period_b": "2024-12"}, "obs": {"ratio": 1.1708, ...}}
{"step": 2, "call": "get_fx_rate", "args": {"currency": "CNY", "on_date": "2024-12-31"}, ...}
{"step": 3, "call": "calculate", ...}
```

**Причина:** `compare_periods` уже дал ratio и delta, но агент не доверил результату и пересчитал через отдельный `get_fx_rate` на 31.12 — получились чуть другие цифры в ответе.

**Что бы исправил:** в описании `compare_periods` добавить «достаточно для ответа о динамике, не вызывай get_fx_rate повторно».

## 4. Вывод

Самый хрупкий элемент — **согласованность данных и дат**. CSV обновляется с лагом, ЦБ иногда недоступен (fallback_csv), а агент не знает, какой месяц «последний актуальный». Из-за этого он дергает `get_inflation` на несуществующие месяцы, потом откатывается назад, тратит лишние шаги и иногда дублирует расчёты, хотя `compare_periods` уже всё посчитал.

Второе слабое место — **промпт**: без явных правил агент на Q7 вызвал `get_key_rate` шесть раз вместо трёх, на Q10 — пересчитал то, что уже было в `compare_periods`.

Лог `trace.jsonl` сильно помог при разборе — без него непонятно, на каком шаге агент «поплыл».
